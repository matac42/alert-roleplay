01 / Nagiosアラートの読み方

Nagiosからのアラートメールには、以下の情報が含まれます。

***** Nagios Alert *****

Notification Type: PROBLEM
Host:             web-01
Service:          CPU Load
State:            CRITICAL
Output:           CRITICAL - load average: 8.52, 6.31, 3.14

各フィールドの意味

  • Notification Type — PROBLEM（障害発生）/ RECOVERY（復旧）
  • Host — アラートが発生したホスト名
  • Service — 監視しているサービス名（CPU Load, HTTP, Disk Usage 等）
  • State — WARNING / CRITICAL / UNKNOWN
  • Output — 監視プラグインの出力（具体的な数値やメッセージ）

02 / 対応フロー全体像

アラートを受けてから完了するまでの基本フローです。

①アラート受信
→
②内容確認
→
③サービス影響確認
→
④SSH接続
→
⑤エビデンス取得
→
⑥原因特定
→
⑦対応実施
→
⑧復旧確認
→
⑨報告

ステップ詳細

  1. アラート受信 — メール/チャットで通知を受ける
  2. 内容確認 — ホスト名・サービス名・状態・Outputを読む
  3. サービス影響確認 — ブラウザやcurlでサイト・APIの表示を確認し、ユーザー影響の有無を把握する
  4. SSH接続 — 対象ホストにSSHでログイン
  5. エビデンス取得 — w, ps, netstat, top, vmstat 等でシステム状態を記録
  6. 原因特定 — 取得したエビデンスから原因を推定
  7. 対応実施 — プロセス停止・サービス再起動・ディスク整理等
  8. 復旧確認 — 対応後にエビデンスを再取得し、正常化を確認
  9. 報告 — 発生時刻・原因・対応内容・現在の状態をまとめて報告
⚠ ③のサービス影響確認はSSHより先に行う。サービスが落ちているかどうかで対応の緊急度・優先度が大きく変わる。影響ありなら即エスカレーションの判断も必要。

03 / SSH接続

監視対象のサーバーに接続するには SSH（Secure Shell）を使います。

基本構文

ssh ユーザー名@ホスト名またはIPアドレス

例

$ ssh ops@192.168.1.10
ops@192.168.1.10's password: ********
ops@web-01:~$ _

ポート指定

SSH のデフォルトポートは 22 ですが、変更されている場合は -p オプションで指定します。

$ ssh -p 2222 ops@192.168.1.10

切断

作業が完了したら exit で切断します。接続したままにしないこと。

ops@web-01:~$ exit
logout
Connection to 192.168.1.10 closed.

04 / エビデンス取得 (1) — w / uptime

サーバーに接続したら、まず w コマンドで全体の負荷を確認します。

w コマンドの出力例

 14:23:01 up 3 days, 7:42,  1 user,  load average: 8.52, 6.31, 3.14
USER     TTY      FROM             LOGIN@   IDLE   JCPU   PCPU WHAT
ops      pts/0    192.168.1.1      14:20    0.00s  0.01s  0.00s w

load average の読み方

  • 8.52 — 直近 1分間の平均
  • 6.31 — 直近 5分間の平均
  • 3.14 — 直近 15分間の平均

load average は実行待ち + I/O待ちプロセスの平均数です。CPUコア数と比較して判断します。

例: 2コアCPUでload 8.52 → 各コアに約4.26の負荷。通常の4倍以上で明らかに過負荷。
⚠ load average にはI/O待ち（D状態）のプロセスも含まれる。load が高い = CPU負荷が高い とは限らない。vmstat の wa 列やプロセスの STAT 列（D=I/O待ち）も合わせて確認すること。
Q. 以下の w 出力を見て、このサーバーの状態として最も適切なものは？
load average: 12.50, 11.80, 4.20（2コアCPU）
load average の3つの値のうち、最も長期的な傾向を示すのは何分平均？

05 / エビデンス取得 (2) — ps auxfww

全プロセスの一覧を確認するコマンドです。CPU・メモリの消費量で原因プロセスを特定します。

コマンド

$ ps auxfww
  • a — 全ユーザーのプロセスを表示
  • u — ユーザー情報付きで表示
  • x — 端末に紐づかないプロセスも表示
  • f — ツリー表示（親子関係がわかる）
  • ww — コマンドラインを省略しない

出力例

USER       PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root         1  0.0  0.1   1840  1200 ?        Ss   Feb27   0:03 /sbin/init
root       892 98.2  0.1   7400  1100 ?        R    14:20   2:45 stress-ng --cpu 2
root       893 97.8  0.1   7400  1080 ?        R    14:20   2:44  \_ [stress-ng-cpu]

上の例では PID 892 が CPU 98.2% を消費しています。stress-ng が原因プロセスとわかります。

重要な列

  • %CPU — CPU使用率（高いものが原因候補）
  • %MEM — メモリ使用率
  • STAT — R=実行中, S=スリープ, Z=ゾンビ, D=IO待ち
  • COMMAND — 実行コマンド（何が動いているかの判断材料）

06 / エビデンス取得 (3) — netstat / ss

ネットワークの接続状態とLISTENポートを確認するコマンドです。

コマンド

$ sudo netstat -antp   # 従来のコマンド（net-tools）
$ sudo ss -antp        # 新しいコマンド（iproute2、推奨）
  • -a — 全ての接続を表示
  • -n — アドレスをIPで表示（名前解決しない）
  • -t — TCP接続のみ
  • -p — プロセス名/PIDを表示
💡 ss は netstat の後継コマンドです。最近のLinuxディストリビューションでは net-tools（netstat）がインストールされていないことがあるため、ss も使えるようにしておくと安心です。オプションはほぼ同じ。
⚠ -p オプションには sudo が必要です。sudo なしだと他ユーザーのプロセス名が表示されません。

出力例

Proto Recv-Q Send-Q Local Address      Foreign Address    State       PID/Program
tcp        0      0 0.0.0.0:22         0.0.0.0:*          LISTEN      142/sshd
tcp        0      0 0.0.0.0:8081       0.0.0.0:*          LISTEN      168/nginx
tcp        0      0 0.0.0.0:5666       0.0.0.0:*          LISTEN      200/nrpe

LISTEN 状態のポートがサービスが待ち受けているポートです。期待するサービスがLISTENしていなければ障害の手がかりになります。

07 / エビデンス取得 (4) — top / vmstat

リアルタイムでシステム状態を監視するコマンドです。

top コマンド

$ top -c -d 1
  • -c — コマンドラインをフル表示
  • -d 1 — 1秒間隔で更新
  • 終了は q キー

CPU使用率順にプロセスがリアルタイムで並ぶため、高負荷の原因を素早く特定できます。

vmstat コマンド

$ vmstat 1
procs ---memory--- ---swap-- -----io---- -system-- ------cpu-----
 r  b  swpd   free   si   so    bi    bo   in   cs us sy id wa st
 2  0     0 112000    0    0   120    80  250  480 95  3  0  2  0

vmstat の重要な列

  • r — 実行待ちプロセス数（CPUの待ち行列）
  • swpd — 使用中のスワップ量（0以外ならメモリ不足の兆候）
  • si / so — スワップイン/アウト（頻発するとパフォーマンス劣化）
  • us — ユーザー空間CPU使用率
  • wa — IO待ちCPU率（高いとディスクがボトルネック）
  • id — アイドル率（0に近いとCPU飽和）
Q. vmstat で wa（IO wait）が高い場合、ボトルネックは何？
上の vmstat 出力で us=95, id=0 です。ボトルネックはどのリソース？

08 / リソース確認 — df / free / du

ディスクとメモリの状態を確認するコマンドです。

df -h（ディスク使用量）

$ df -h
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda1       2.0G  680M  1.2G  36% /
tmpfs            64M  4.0K   64M   0% /tmp
  • -h — 人間が読みやすい単位（MB/GB）で表示
  • Use% が 90% 以上なら危険。100% になるとサービスが停止する可能性あり

free -h（メモリ使用量）

$ free -h
              total    used    free  shared  buff/cache  available
Mem:          460M    128M    112M     4M       220M      312M
Swap:           0B      0B      0B
  • available（利用可能メモリ）を見る。free ではなく available が実際に使える量
  • Swap の used が大きい場合はメモリ不足の兆候

du -sh（ディレクトリサイズ）

$ du -sh /var/log/*
120M    /var/log/messages
45M     /var/log/nginx

ディスク容量不足のとき、どのディレクトリが容量を消費しているか特定できます。

09 / サービス管理 — systemctl / journalctl

サービスの状態確認・起動・再起動と、ログの確認方法です。

systemctl — サービス管理

$ systemctl status nginx       # 状態確認
$ sudo systemctl start nginx   # 起動
$ sudo systemctl restart nginx # 再起動
$ sudo systemctl stop nginx    # 停止
💡 status は sudo なしで実行可能。start / restart / stop には sudo が必要です。

journalctl — ログ確認

$ journalctl -u nginx --no-pager -n 50  # 直近50行
$ journalctl -u nginx --since "1 hour ago"
  • -u サービス名 — 特定サービスのログのみ
  • -n 数字 — 直近N行を表示
  • --since — 時間指定でフィルタ

10 / 対応アクション — kill / rm / sudo

原因を特定した後の対応コマンドです。

kill — プロセス停止

$ sudo kill PID         # 正常終了シグナル (SIGTERM)
$ sudo kill -9 PID      # 強制終了 (SIGKILL)
  • まず kill PID で正常終了を試みる
  • 応答しない場合のみ kill -9 で強制終了
  • 他ユーザーのプロセスを停止するには sudo が必要
⚠ kill する前に必ず ps で PID と COMMAND を確認すること。誤ったプロセスを停止すると障害が悪化します。

ディスク容量対応 — ログファイル

ログが肥大化してディスクを圧迫している場合、まず logrotate の設定を確認します。

$ cat /etc/logrotate.d/nginx       # ログローテート設定の確認
$ sudo logrotate -f /etc/logrotate.d/nginx # 手動でローテート実行

logrotate で対処できない緊急時のみ、以下の方法を使います。

$ sudo truncate -s 0 /var/log/app/huge.log # ファイルを空にする（FDが保持される）
$ sudo rm /var/log/app/huge.log             # ファイル削除（FDが残るとディスクが解放されない場合あり）
💡 rm でファイルを削除しても、プロセスがファイルを開いたままだとディスク容量は解放されません。truncate の方が安全です。根本対策は logrotate の設定見直し。

sudo の注意点

  • 必要なときだけ sudo を使う（最小権限の原則）
  • sudo を付けたコマンドは root 権限で実行される
  • root で直接操作するよりも sudo で都度昇格する方が安全

11 / エスカレーション判断

一次対応者として「自分で対応する」か「上位者に引き継ぐ（エスカレーション）」かの判断基準です。

自分で対応できるケース

  • 原因が明確で、手順書（ランブック）に記載がある
  • 不正なプロセスの停止（stress-ng, マイニングプロセス等）
  • ログファイル肥大化によるディスク圧迫 → ログクリーンアップ
  • サービスの単純な再起動で復旧する問題

エスカレーションすべきケース

  • 原因が特定できない / 複数の要因が絡んでいる
  • 対応に影響範囲の判断が必要（DB操作、設定変更等）
  • ハードウェア障害の疑い（ディスクエラー、メモリエラー）
  • セキュリティインシデントの疑い（不審なプロセス、不正アクセス）
  • サービス再起動で復旧しない
  • 自分の権限では対応できない操作が必要
💡 迷ったらエスカレーション。「報告が遅れた」方が「誤った対応をした」より遥かに深刻です。
